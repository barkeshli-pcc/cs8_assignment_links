CS — Data Structures · Project: Table

A component of the final SQL Database project

Before You Start: System-Wide Constraints

These two rules apply to every project in this sequence, not just this one:

  • No STL containers or algorithms beyond string and vector. Every ordered or lookup structure — B+ tree, Map, Multimap, Stack, Queue, Set — is one you build yourself. Do not substitute std::map, std::set, std::stack, std::queue, std::find, or any other STL container or algorithm for functionality this system is asking you to implement.
  • No sequential search, anywhere, for any reason. Any time your code needs to look something up — a keyword, a field name, a table name, a stored value — that lookup must go through a B+ tree-backed Map or Multimap. A linear scan is not acceptable, no matter how short the list being scanned is or how the scan is written.

1. Overview

You've already built the pieces that store data on disk (FileRecord) and the pieces that turn raw SQL text into a structured request (the Parser). This project builds the piece in between: a Table that actually holds rows of data and can answer questions about them — insert a row, print every row, or select rows matching a condition, including compound conditions with AND, OR, and parentheses.

Answering a compound condition efficiently takes two supporting pieces, which you'll build alongside Table:

  • A small family of condition token types (a field name or value, a relational operator, a logical operator, parentheses) that know how to evaluate themselves against the table's data.
  • A ShuntingYard that takes a condition written in the order a person would type it (infix) and reorders it into the order it should be evaluated in (postfix), the same way a calculator turns 2 + 3 * 4 into an evaluation order that respects * binding tighter than +.

None of this touches SQL syntax directly — Table never sees the string "select ... where ...". It receives a table name, a list of fields, and a condition already broken into tokens. Turning SQL text into that form is the Parser's job, not this one.

2. What It Must Accept / Do

Condition tokens

A condition is built out of a small set of token types:

  • A string token — a bare field name (age) or a literal value (20, Smith). The token type itself doesn't distinguish a field name from a value; that distinction is made by position (an operator's left operand is always a field name, its right operand always a value).
  • A relational operator — one of =, <, >, <=, >=. Applied to a field name and a value, it produces the set of record numbers whose stored value for that field satisfies the comparison.
  • A logical operator — and or or. Applied to two already-evaluated sets of record numbers, it produces their intersection (and) or union (or).
  • Parentheses — used only to override default evaluation order; they carry no meaning once that order has been determined.

Operator precedence (needed to evaluate a condition correctly when parentheses are absent): relational operators bind tighter than logical operators, and among logical operators, and binds tighter than or. So age < 20 and lname = Smith or age > 65 is evaluated as (age < 20 and lname = Smith) or (age > 65), not left-to-right.

Not required: the != operator and unary not are not exercised by this project's tests — you do not need to implement evaluation for them.

ShuntingYard

Given a condition as a flat list of tokens in infix order — either already-typed condition tokens, or plain strings like {"age", ">", "18", "and", "lname", "=", "Smith"} — produce the same tokens reordered into postfix order, ready to be evaluated left to right with a stack. Parentheses in the input must correctly override the default precedence described above.

Table

A Table must support:

  • Creating a new table, given a name and a list of field names. This persists both the (currently empty) row storage and the field-name list, so the table can be reopened later.
  • Loading an existing table, given only a name. This reads back the persisted field-name list and reconstructs whatever in-memory structures are needed to answer queries efficiently, from the persisted rows alone — a freshly loaded table must be immediately queryable without requiring every row to be re-inserted.
  • Inserting a row, given a list of field values in field order. The row is persisted, and becomes visible to subsequent selects on this table object.
  • Selecting rows, in three forms, all returning a new table containing only the matching rows, projected onto a requested list of fields:
    • A single condition given directly as (field, operator, value) — no token machinery involved.
    • A condition given as an already-built postfix sequence of condition tokens (as described above).
    • A condition given as a flat infix list — plain strings, with parentheses allowed — which must be converted to postfix internally before being evaluated.
  • Reporting which original rows matched the most recent select, by record number (referring to the row's position in the table that was searched, not in the result table).
  • Printing a table: its name, its row count, a header of field names, and then every row, in the same tabular, fixed-width-column style used for individual records elsewhere in this system.

Storing comparable values: every field value is stored and compared as text, including values that look numeric. Relational comparisons (<, >, <=, >=) are performed on that text directly, not converted to a number first — so "9" compares as greater than "18". If a field is meant to be compared as a number (like an age), the values you insert must all be padded to the same width (e.g., always two digits) so that text comparison and numeric comparison agree.

3. Where It Fits

Parser (produces a command + field list + flat condition tokens)
   |
   v
Table  --  uses FileRecord to store/retrieve rows
       --  uses ShuntingYard + condition tokens to evaluate WHERE clauses
   |
   v
Result printed to screen

Table doesn't parse SQL, doesn't know keywords like select or where, and doesn't validate that a condition's field names actually exist before evaluating — it trusts the field names and values it's given.

4. Input/Output Contract

Creating a table and inserting rows:

Table t("employee", {"fname", "dept", "level"});
t.insert_into({"Priya", "eng", "05"});
t.insert_into({"Marco", "sales", "12"});
t.insert_into({"Priya", "sales", "12"});
t.insert_into({"Wei", "eng", "09"});

Printing a table shows its name, its row count, a header row, then each stored row, right-justified in fixed-width columns:

Table name: employee, records: 4
                   record                    fname                     dept                    level

                        0                    Priya                      eng                       05
                        1                    Marco                    sales                       12
                        2                    Priya                    sales                       12
                        3                      Wei                      eng                       09

A single-condition select projects onto the requested fields and returns a new table of just the matching rows:

t.select({"fname", "level"}, "dept", "=", "sales");
Table name: _select_table_1, records: 2
                   record                    fname                    level

                        0                    Marco                       12
                        1                    Priya                       12

After any select call, the record numbers of the matching rows — as they were numbered in the table that was searched, not in the result — are available separately; their order reflects how the underlying lookup encountered them and is not guaranteed to be sorted by record number.

A compound condition, given as a flat infix list with parentheses, is also accepted directly:

vector<string> condition = {"(", "level", ">", "10", "and", "dept", "=", "sales", ")", "or", "fname", "=", "Wei"};
t.select({"fname", "dept", "level"}, condition);
Table name: _select_table_2, records: 3
                   record                    fname                     dept                    level

                        0                    Marco                    sales                       12
                        1                    Priya                    sales                       12
                        2                      Wei                      eng                       09

5. Design

Two ideas make compound conditions practical:

  • Per-field indexing. Rather than scanning every row to answer dept = "sales" or level > "10", maintain, for each field, a structure that maps a value to the record numbers holding that value — built once when a table is loaded or created, and kept up to date as rows are inserted. This is what makes a single-condition select answer directly from the index rather than reading the whole table.
  • Evaluating postfix with a stack. A relational operator consumes a field name and a value and produces a set of record numbers (using that field's index). A logical operator consumes two sets of record numbers and produces their intersection or union. Walking a postfix token sequence left to right with a single stack — pushing operands, and replacing an operator's operands with its result — evaluates any condition, however deeply nested, without needing to know its structure in advance.

Think of "evaluate this operator" as something each token type knows how to do to itself, given the data (or the previously-computed operand sets) it needs — rather than one large function with a branch for every operator.

You are free to choose how a "set of record numbers" is represented internally, and how set-combination (intersection/union) is implemented, as long as the observable results match Section 4.

6. What You Are Building

  • Condition token types for: a bare string (field name or value), each of the five required relational operators, both logical operators, and parentheses.
  • Correct operator precedence when a condition has no parentheses: relational tighter than logical, and tighter than or.
  • A ShuntingYard that converts a flat infix token/string list into correct postfix order, honoring explicit parentheses.
  • A Table constructible either as brand-new (name + field list) or as a load of an existing table (name only), immediately queryable in both cases.
  • insert_into that persists a row and keeps per-field indexing consistent with what's on disk.
  • All three select forms described in Section 2, each returning a new table of matching rows projected onto the requested fields.
  • Correct reporting of which original record numbers matched the most recent select.
  • Printing that matches the header-plus-rows, fixed-width-column format shown in Section 4.
