System Overview — read this before starting the first project below

What You're Building

Over the semester, you're building a working, interactive database engine that responds to SQL-like commands — make table, insert into, and select — typed as plain text. It stores its data in binary files on disk, so it remembers everything between separate runs of the program, the same way a real database does.

Nothing in this system is one big program. It's built as a sequence of small, separately-tested pieces, each depending only on the pieces before it. This page exists to show you the whole shape before you dive into the first piece.

The Pieces, in Build Order

Earlier in the semester you built the foundation this all sits on:

  • B+ Tree — the ordered, disk-friendly tree structure everything else indexes through.
  • Map / Multimap — key-value and key-multivalue containers built on top of your B+ Tree.
  • Tokenizer — splits a raw string into a list of tokens.
  • Stack / Queue — used by expression evaluation later in the sequence.

The four projects that follow — the ones with their own detailed writeups — build the database itself, in this order:

  1. Parser — turns a raw SQL command string into a structured parse tree (which command, which table, which fields, which condition), using a finite state machine over the tokens your tokenizer produces.
  2. Binary File Records (FileRecord) — stores a single table row as a fixed-size binary "envelope" on disk, addressable directly by row number.
  3. Table — manages one table's rows and per-field indices; evaluates conditions (including compound AND/OR/parenthesized ones) via a small typed-token hierarchy and a shunting-yard infix-to-postfix conversion.
  4. SQL — ties Parser and Table together: takes one command string, dispatches it, and remembers which tables exist across separate runs of the program.

How a Command Flows Through the System

User types a SQL command string
        |
        v
     Parser   -- raw string -> parse tree
        |
        v
      SQL     -- reads the parse tree, calls the matching Table operation
        |
        v
     Table    -- stores rows (via FileRecord), evaluates conditions
        |
        v
Result table printed to screen

Each arrow above is a boundary between two of your own classes, and each class only knows about the layer directly below it. The Parser never sees a Table. The Table never sees a parse tree. This separation is deliberate — it's what lets each piece be built and tested independently, and it's why the four writeups can be read and built one at a time.

Before You Start: System-Wide Constraints

These two rules apply to every project in this sequence:

  • No STL containers or algorithms beyond string and vector. Every ordered or lookup structure — B+ tree, Map, Multimap, Stack, Queue, Set — is one you build yourself. Do not substitute std::map, std::set, std::stack, std::queue, std::find, or any other STL container or algorithm for functionality this system is asking you to implement.
  • No sequential search, anywhere, for any reason. Any time your code needs to look something up — a keyword, a field name, a table name, a stored value — that lookup must go through a B+ tree-backed Map or Multimap. A linear scan is not acceptable, no matter how short the list being scanned is or how the scan is written.

How to Use the Four Project Writeups

Each of the four projects (Parser, FileRecord, Table, SQL) has its own writeup with a full specification, worked examples, and a checklist of what you're building. Read them in the order listed above — each one assumes the previous ones are already done and working.
