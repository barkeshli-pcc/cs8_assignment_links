Before You Start: System-Wide Constraints

These two rules apply to every project in this sequence, not just this one:

  • No STL containers or algorithms beyond string and vector. Every ordered or lookup structure — B+ tree, Map, Multimap, Stack, Queue, Set — is one you build yourself. Do not substitute std::map, std::set, std::stack, std::queue, std::find, or any other STL container or algorithm for functionality this system is asking you to implement.
  • No sequential search, anywhere, for any reason. Any time your code needs to look something up — a keyword, a field name, a table name, a stored value — that lookup must go through a B+ tree-backed Map or Multimap. A linear scan is not acceptable, no matter how short the list being scanned is or how the scan is written.

1. Overview

Earlier in the semester you wrote a small program that stores single strings in a binary file — one Record per fixed-size slot, written and read back by slot number. That version only ever held one piece of information per record: a single word or name.

A real table row is not one string — it's several: a name, an ID, an age, a department, however many columns the table has. This project generalizes your Record into a FileRecord: a single binary "envelope" that holds an entire row's worth of string fields, still stored and retrieved as a fixed-size slot in a binary file. You will also factor out the general-purpose file-handling helpers you already have into their own reusable module, and add two new ones for saving and loading simple lists of strings as plain text.

This is infrastructure, not database logic. Nothing here knows about SQL, table schemas, or column types — it only knows how to store and retrieve fixed-size binary rows, and how to persist small lists of text. The Table class you'll build afterward is the first thing that will actually use this to store real rows on disk.

2. What It Must Accept / Do

FileRecord

A FileRecord holds a fixed maximum number of fields, each field itself a fixed-length string. Concretely:

  • A FileRecord can hold up to 50 fields. Each field can hold up to 49 characters (plus its terminator). You are not required to guard against a caller exceeding either limit — that's out of scope for this project.
  • A FileRecord can be constructed in two ways:
    • From a vector<string> — one entry per field, in order. This is the form used to build a row of real data.
    • From a single long (defaulting to 0) — this produces a record whose first field holds that number (as text) and every other field empty. This form exists to create an empty placeholder object to be filled in by a subsequent read(), not to hold real row data.
  • rec_no() reads field 0 and returns it as a long. This only makes sense when field 0 actually contains numeric text — if you call it on a record whose field 0 holds something like a name, the result is undefined. It is the caller's responsibility to only call rec_no() when that assumption holds.

Utilities

Alongside FileRecord, provide a small set of file-handling helpers, usable independently of any specific record type:

  • file_exists(filename) — true if the file can currently be opened, false otherwise.
  • open_fileW(f, filename) — open filename for output only, in binary mode, creating or truncating it. Throws on failure.
  • open_fileRW(f, filename) — open filename for combined reading and writing, in binary mode. If the file does not already exist, create it first.
    Behavioral note: when the file has to be created, this call leaves it open for output only — not for reading yet. If you need both read and write access on a brand-new file, call it a second time (the file now exists, so the second call opens it with both read and write access). Design your callers with this in mind rather than assuming a single call always yields a read/write-capable stream.
  • read_info(filename, v) — open filename as plain text, read it line by line, and append each line (in order) onto the end of the given vector<string>. Return false if the file cannot be opened; true otherwise, even if the file is empty (in which case nothing is appended).
  • write_info(filename, v) — write the given vector<string> to filename as plain text, one entry per line, overwriting anything already there. Return false if the file cannot be opened for writing.
  • show_vector(v) — print every element of a vector to standard output, separated by |, for debugging. Works for any element type that supports <<.

FileRecord and the utilities module know nothing about column names, column types, or SQL. They don't validate what's inside a field, and they don't know how many fields a "real" table row is supposed to have beyond the fixed maximum. That knowledge belongs to the Table class layer above this one.

4. Input/Output Contract

Building a record from field values and writing it:

FileRecord r(vector<string>{"Ada Lovelace", "101"});
long slot = r.write(outfile);

write() writes the entire fixed-size record — all fields, not just the ones you filled in — to the stream at the stream's current write position. It does not seek to the end of the file itself; it assumes the stream is already positioned where you want this record to land (for sequential writes to a freshly opened output file, that position naturally advances after each call). It returns the slot number for this record, computed as position ÷ record size — so writing three records in a row to an empty file yields slot numbers 0, 1, 2.

Reading a specific slot back — including one you haven't read before, out of order:

FileRecord r2;
r2.read(infile, 1);   // jump directly to slot 1, no need to read slot 0 first
cout << r2 << endl;

Given a slot number, read() seeks directly to that slot's byte offset (slot number × record size) and reads one full record's worth of bytes, regardless of what was previously in memory. It returns the number of bytes actually read. At or past the end of the file, this is less than a full record's worth — as low as 0 — which is how a caller detects there's no more data to read, without needing to know how many records the file holds in advance.

Printing a record (via operator<<) prints each field, left to right, right-justified in a fixed 25-character column, stopping at the first field whose stored value is empty:

              Ada Lovelace                      101
                Alan Turing                      102

Fields after the first empty one are never printed, no matter what bytes happen to be sitting in them. A record built from a two-entry vector prints exactly two fields — never more, even though the underlying storage physically holds up to 50.

Saving and loading a simple text list — for example, a table's field names:

write_info("student.fields", vector<string>{"lname", "fname", "age"});
...
vector<string> fields;
read_info("student.fields", fields);   // fields is now {"lname", "fname", "age"}

5. Design

The core idea is a fixed-size binary record: every record, no matter how many fields it actually uses, occupies exactly the same number of bytes on disk. That's what makes direct access possible — to fetch record number k, you compute k × (record size) and seek straight there, with no need to scan through the file or know where previous records ended. Contrast this with your original single-field Record: that class demonstrated the fixed-slot idea with one string per slot; FileRecord applies the same idea to a slot that holds several strings at once.

Concretely, you need a way to store a bounded, ordered collection of fixed-length strings inside a single fixed-size structure, such that:

  • The whole structure has one constant size in bytes, known ahead of time, so slot arithmetic always lands correctly.
  • An unused field is distinguishable from a used one, so printing (and later, the Table class's row-parsing) knows where real data ends.

How you organize the fields inside that fixed structure — and how you decide "this field is empty" — is your design choice to make; the contract in Section 4 is what has to hold true regardless of how you build it.

For the utilities module: this is mostly relocating helpers you've already written and tested in your primitive version, plus adding the two text list functions. No new data structure concepts are required here — read_info/write_info are straightforward line-oriented text I/O.

6. What You Are Building

  • A FileRecord type constructible from a vector<string>, holding up to 50 fields of up to 49 characters each.
  • A FileRecord constructible from a single long, for use as an empty placeholder to be filled by read().
  • write() that writes the full fixed-size record at the stream's current position and returns the resulting slot number.
  • read() that seeks directly to a given slot number and returns the number of bytes actually read, including partial/zero reads at end of file.
  • rec_no() that returns field 0 interpreted as a number.
  • Printing that stops at the first empty field and right-justifies each printed field in a 25-character column.
  • A separate, reusable utilities module containing file_exists, open_fileW, open_fileRW (matching the behavior described in Section 2, including the create-then-reopen note), read_info, write_info, and show_vector.
